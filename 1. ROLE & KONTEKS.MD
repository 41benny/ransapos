# MODUL & FUNGSI INTI

## 1. POS (FRONTLINE / KASIR)

Fitur utama:
- Input order:
  - Pilih outlet.
  - Pilih produk (bisa by kategori, search).
  - Atur kuantitas, harga, diskon item / diskon global.
- Pembayaran:
  - Metode: cash, QRIS, e-wallet, transfer (bisa dibuat master `payment_methods`).
  - Menyimpan detail pembayaran di tabel `payments`.
- Struk:
  - Print struk thermal (minimal layout).
  - Nomor transaksi / nomor struk unik per outlet.
- Shift Kasir:
  - Buka shift (saldo awal kas).
  - Tutup shift (rekap penjualan, kas fisik, selisih).
  - Tabel: `cash_sessions`.

Tabel inti untuk POS:
- `sales`
- `sale_items`
- `payments`
- `cash_sessions`

Aturan umum:
- Setiap penjualan yang mengurangi stok wajib memicu **mutasi stok**.

## 2. BACK OFFICE (ADMIN PANEL)

### 2.1 Master Data
- `outlets` (nama, alamat, aktif/tidak).
- `users` + `roles` / `role_user` atau pakai Laravel Permission (boleh simple dulu).
- `products`:
  - nama, sku, kategori, satuan, harga jual default.
- `product_categories`
- `suppliers`

### 2.2 Inventory / Stok
- Tabel stok per outlet:
  - `stocks`: stok per produk per outlet.
- Mutasi stok:
  - `stock_mutations`: 
    - `product_id`
    - `outlet_id`
    - `mutation_type` (in / out / adjustment / transfer_in / transfer_out)
    - `quantity`
    - `reference_type` (purchase / sale / stock_opname / transfer)
    - `reference_id`
    - `mutation_date`
- Purchase:
  - `purchases`
  - `purchase_items`
  - Setiap `purchase` → stok bertambah via `stock_mutations`.

### 2.3 Laporan
Minimal laporan:
- Laporan penjualan per:
  - Tanggal range.
  - Outlet.
  - Produk.
- Laporan stok:
  - Stok akhir per outlet.
  - Mutasi stok per produk dalam periode tertentu.
- Rekap shift kasir:
  - Total omset.
  - Total kas masuk / non-kas.
  - Selisih kas.

# ARSITEKTUR & PEMISAHAN LOGIC

## Routing & Area
- Gunakan prefix dan middleware yang jelas.

Contoh:
- Back Office:
  - Prefix: `/admin`
  - Middleware: `auth`, `role:admin|manager`
- POS:
  - Prefix: `/pos`
  - Middleware: `auth`, `role:kasir|admin`
  - Layout Blade khusus (fullscreen, tombol besar).

## Layering
- Controller tipis: hanya handle Request & Response.
- Business logic kritikal:
  - Ditaruh di **Service class** (misal `SaleService`, `StockService`).
- Validasi:
  - Wajib pakai **Form Request**.

Contoh alur saat transaksi POS:
1. User input order → request ke `POS\SaleController@store`.
2. Controller validasi via `StoreSaleRequest`.
3. Controller memanggil `SaleService::createSale(...)`.
4. `SaleService`:
   - Buat `sales` & `sale_items`.
   - Catat `payments`.
   - Trigger `StockService::applySaleMutation(...)` untuk membuat `stock_mutations` dan update `stocks`.
5. Return response ke POS view.
